# Reflecting on Software Engineering

I signed up for this course thinking I would mainly learn how to build websites. And I did. I can now make a Next.js app, write TypeScript components, work with Prisma, and deploy a project online. But looking back, the most useful things I learned were not tied to any single technology. The skills I’ll probably keep using are the ones that apply to almost any software project: organizing team work, keeping code consistent, and solving problems with reliable design approaches. Three ideas especially stood out to me: Agile Project Management, Configuration Management, and Design Patterns.

Agile Project Management is all about building projects in smaller steps instead of trying to plan everything perfectly from the start. In this class, we used Issue Driven Project Management, where every task had to be tied to a clearly defined issue before anyone started working on it. Each issue had a description, an assigned person, and a way to track progress. Working on StudyViser this way was a new approach for me. Instead of vague plans and agreeements, everyone knew what should be done and who was doing it. I also realized this approach works outside software too. Researchers, architects, or even event organizers could benefit from breaking big goals into smaller tasks and reviewing progress regularly.

Another important concept was Configuration Management, which focuses on keeping software stable and consistent over time. In this course, we used Git, GitHub, and tools like `package.json` to make sure everyone on the team could work in the same environment without constantly breaking the project. Before this class, I never really thought about how much effort goes into maintaining a project beyond just writing code. I learned that software engineering is also about making systems reliable, especially when multiple people are working together.

The third concept that stood out to me was Design Patterns. Before this class, I mostly wrote code just to make things work. I did not think much about how the structure of the code could be reused or improved. Learning patterns like Model-View-Controller helped me understand how larger systems are organized and why certain structures appear over and over again in software. StudyViser itself followed a structure similar to MVC, with Prisma handling data, React components handling presentation, and server actions connecting everything. What I found interesting is that many of these patterns are decades old, but are still reliable.

Overall, this course gave me more than web development skills. It changed how I think about software engineering as a whole. Agile Project Management taught me how to organize work with a team, Configuration Management showed me how to keep projects reliable, and Design Patterns gave me a better way to think about structuring code. Even if the technologies I use change in the future, I think these ideas are the ones I’ll keep coming back to.
